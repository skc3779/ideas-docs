# PRD 기반 실행 프롬프트 (트렌드 분석 -> 아이디어 제안)
> version 1

## 1) 목적
현재 시점의 RSS FEED 트렌드를 수집/요약/분석하여, 대중성이 높은 아이디어 4가지를 제안한다.
또한 각 아이디어별 상세 실행 문서를 별도 파일로 생성한다.

## 2) 최종 산출물

### 메인 결과 파일
- 파일명 규칙: `YYYY년MM월DD일_HH24MMSS.md`
- 포함 내용:
  - 트렌드 요약
  - 아이디어 4개
  - 각 아이디어의 실현 가능성 5점 척도 평가
  - 평가 근거(참고한 RSS FEED 제목, 링크)

### 아이디어 상세 파일 (아이디어별 1개)
- 파일명 규칙: `YYYY년MM월DD일_HH24MMSS_[제안번호].md`
- 예시:
  - `2026년04월18일_121011_1.md`
  - `2026년04월18일_121011_2.md`
- 포함 내용:
  - 제안 범위의 카테고리 영역
  - 실행 방안(깊이 있게)
  - 평가 근거
  - 참고 RSS FEED의 제목, 링크, 요약정보

### 중간 데이터 파일
- 1차 수집 파일: `trends_data_1차_[일시].json`
  - 수집항목: 제목, 링크, 원문링크(해당 시), 출처도메인(해당 시)
- 2차 수집 파일: `trends_data_2차_[일시].json`
  - 수집항목: 제목, 링크, 요약정보

## 5) 수집 및 분석 절차

### Step 1. 1차 수집 (RSS 원문 기준)
- 공통 원칙: 각 FEED에서 최신 3개 수집
- 저장 파일: `trends_data_1차_[일시].json`
- 수집항목: 제목, 링크, 원문링크(해당 시)

#### 소스별 수집 규칙
- Google 트렌드:
  - 제목: `title + " : " + news_item_title`
  - 링크: `news_item_url`
- Google 전체/경제/과학/AI 뉴스:
  - 제목: `title`
  - 링크: `link` (Google News RSS 링크)
  - 원문링크: 아래 우선순위로 실제 기사 URL 추출
    1. RSS 원문 XML의 `description` 내부 외부 기사 URL
    2. RSS 원문 XML의 `source[url]`
    3. Google News 링크 응답의 최종 도착 URL(리다이렉트 체인)
    4. 제목 + 언론사명 기반 재검색으로 후보 URL 복원
  - 비고: 원문링크가 `news.google.com` 도메인이면 추출 실패로 간주하며, 1차 수집 완료 전에 반드시 재시도
  - 추가 저장: `출처도메인`(RSS `source[url]` 기반) 필드를 함께 저장
- geeknews:
  - 제목: `title`
  - 링크: `entry > link[href]`
- TechCrunch AI / OpenAI Blog:
  - 제목: `title`
  - 링크: `link`

### Step 2. 2차 수집 (상세 페이지 방문 요약)
- 입력: 1차 수집 JSON
- 저장 파일: `trends_data_2차_[일시].json`
- 수집항목: 제목, 링크, 요약정보
- 방법:
  - Google 전체/경제/과학/AI 뉴스는 `원문링크`를 우선 방문
  - `원문링크`가 없거나 접근 실패 시 `link` 방문
  - Google News 중간 페이지 텍스트(예: "Google News")만 추출되면 실패로 간주하고 다른 원문 후보 URL 재시도
  - "Enable JavaScript and cookies to continue", "접속 실패", 빈 본문 등 저품질 텍스트는 요약 성공으로 인정하지 않음
  - 핵심 내용 요약(사건/기술/시장 반응 중심)

#### Google 뉴스 리다이렉트 처리 규칙(필수)
- 대상: `news.google.com/rss/articles/...` 형식 링크
- 처리 순서:
  1. RSS 원문의 `description` HTML에서 외부 기사 URL 추출
  2. HTTP 요청 시 리다이렉트 체인을 추적하여 최종 도착 URL 확보
  3. 최종 URL이 여전히 `news.google.com`이면 본문 요약 수집 실패로 처리하고 대체 URL 재탐색
  4. 대체 URL은 `제목 + 언론사` 검색으로 3개 이상 확보 후, 도메인 일치/제목 유사도 기준으로 1개 선택
  5. 선택 URL 본문 추출 결과가 품질 기준 미달이면 후보 2순위/3순위 순차 재시도
- 저장 규칙:
  - `link`: RSS 원본 링크 유지
  - `원문링크`: 실제 기사 URL 저장
  - `summary`: 실제 기사 본문 기반 요약만 허용

#### 원문링크 유효성 검증 규칙(필수)
- 아래 조건을 모두 만족해야 `원문링크`로 확정한다.
  1. `news.google.com` 도메인이 아님
  2. 도메인이 기사 출처(`title`의 언론사명, RSS `source[url]`)와 일치하거나 합리적으로 동일 계열임
  3. URL 경로가 기사형 패턴임(예: `/news/`, `/article/`, `/view/`, `/articles/`, 날짜 경로, slug 포함)
  4. 홈페이지 루트(`https://domain.com/`) 또는 섹션 메인(`/news`, `/sports` 등)만 가리키지 않음
  5. 페이지 본문과 제목의 핵심 키워드가 최소 2개 이상 일치
- 아래 조건 중 하나라도 해당하면 `원문링크` 후보에서 즉시 제외한다.
  - 검색 결과가 원 기사와 무관한 위키/사전/커뮤니티/소프트웨어 소개 페이지
  - 원 기사와 무관한 브랜드 메인 사이트(예: 기업 소개 메인)
  - 로그인이 필요한 랜딩/봇 차단/리다이렉트 전용 페이지

#### 소스별 도메인 가드레일(필수)
- Google 트렌드:
  - `ht:news_item_url`을 최우선 사용한다.
  - `title`의 키워드(인물/팀/이슈)와 본문 키워드 불일치 시 재선정한다.
- Google 전체/경제/과학/AI 뉴스:
  - `source[url]`는 "후보 도메인"으로만 사용하고, 기사 URL로 간주하지 않는다.
  - 반드시 기사 상세 경로까지 확인 후 확정한다.
- OpenAI Blog:
  - 허용 도메인을 `openai.com`으로 제한한다.
  - `openai.com` 외 도메인은 검색 결과라도 원문링크로 저장하지 않는다.
- TechCrunch AI:
  - 허용 도메인을 `techcrunch.com`으로 제한한다.

#### 원문링크 점수화 선택 방식(권장)
- 후보 URL마다 점수를 계산해 최고점 1개를 선택한다.
  - 도메인 일치: +40
  - 기사형 경로: +30
  - 제목 키워드 2개 이상 일치: +20
  - 발행일/작성일 근접: +10
  - 루트/섹션 메인 페이지: -50
  - 무관 도메인/저품질 페이지: -80
- 최종 점수가 60점 미만이면 `원문링크 확정 실패`로 처리하고 재탐색한다.

#### 요약 품질 기준(필수)
- 성공 기준:
  - 본문 길이 120자 이상
  - 의미 있는 문장 2개 이상
  - 중간 페이지/차단 안내 문구가 아님
  - 내비게이션/카테고리/로그인 안내 텍스트 비율이 50% 미만
- 실패 기준:
  - "Google News", "Enable JavaScript and cookies to continue", "접근 실패", "본문 추출 실패" 등 고정 실패 문구
  - 본문 길이 120자 미만
  - 메뉴/카테고리/푸터 나열이 본문 대부분을 차지하는 경우
- 실패 시 조치:
  - 원문 후보 URL 재탐색 -> 재요청 -> 재요약을 최대 3회 수행
  - 3회 실패 시 `요약정보`에 실패 사유와 시도 URL 목록 기록

#### 2차 수집 전 전처리(필수)
- `원문링크`가 아래 중 하나면 2차 수집 전에 강제 재선정한다.
  - 도메인만 있는 URL (`https://www.khan.co.kr` 형태)
  - 루트/섹션 메인 URL
  - 허용 도메인 정책 위반 URL
- 재선정 후에도 통과하지 못하면 해당 항목은 `요약 보류`로 기록하고, 근거(실패 사유/시도 URL)를 남긴다.

#### 재탐색 구현 가이드(권장)
- 검색 쿼리 우선순위:
  1. `site:{출처도메인} {제목(언론사 suffix 제거)}`
  2. `{제목(언론사 suffix 제거)}` 일반 검색
- 후보 선택 규칙:
  - 먼저 `site:` 제한 결과에서 검증 통과 URL을 선택
  - 없으면 일반 검색 결과 중 검증 통과 URL 선택
  - 검증 통과 URL이 없으면 요약 보류 처리

### Step 3. 심층 분석
- 대상: 2차 수집 JSON
- 분석 포인트:
  - 반복 출현 키워드
  - 수요 신호(불편/관심/지출 의향)
  - 실행 가능성(기술/비용/시장 진입 난이도)

### Step 4. 아이디어 제안
- 개수: 4개
- 조건:
  - 서로 다른 수요/문제를 반영
  - 제안 범위(4개 영역) 중 카테고리 명시
  - 트렌드 근거 기반으로 작성

### Step 5. 상세 실행 문서 작성
- 아이디어별 별도 파일 생성
- 실행 계획을 구체적으로 기술:
  - 문제 정의
  - 타겟 사용자
  - 핵심 기능
  - MVP 범위
  - 수익 모델
  - Go-To-Market
  - 개발/운영 로드맵
  - 리스크 및 대응

## 6) 평가 기준 (5점 척도)
아이디어별로 실현 가능성을 1~5점으로 평가하고 근거를 작성한다.

- 5점: 즉시 실행 가능, 기술/시장 검증 강함
- 4점: 실행 가능, 일부 검증 필요
- 3점: 가능성 보통, 핵심 가정 검증 필요
- 2점: 리스크 높음, 선행 조건 다수 필요
- 1점: 현재 시점 실행 어려움

각 아이디어에 반드시 포함:
- 점수
- 근거 요약
- 참고 RSS FEED 제목 + 링크

## 7) 출력 형식 템플릿

### 7-1. 메인 파일 템플릿
1. 수집 개요 (일시, 수집 소스, 건수)
2. 트렌드 종합 요약
3. 아이디어 제안 (4개)
4. 아이디어별 실현 가능성 평가표
5. 참고 RSS FEED 목록 (제목, 링크)

### 7-2. 상세 파일 템플릿 (아이디어별)
1. 아이디어명
2. 제안 범위 및 카테고리
3. 해결 문제와 사용자 페르소나

## 8) 품질 체크리스트
- 아이디어는 정확히 4개인가?
- 각 아이디어에 카테고리 영역이 명시되었는가?
- 각 아이디어에 5점 척도 평가와 근거가 있는가?
- 근거에 RSS FEED 제목/링크가 포함되었는가?
- 메인 파일 + 상세 파일(4개) 생성 규칙을 지켰는가?
- 상세 파일에 실행 방안이 충분히 깊이 있게 작성되었는가?
- Google 전체/경제/과학/AI 뉴스 요약이 "Google News" 고정값으로 남지 않았는가?
- Google 전체/경제/과학/AI 뉴스 항목에 `원문링크`가 저장되었는가?
- Google 전체/경제/과학/AI 뉴스의 `원문링크` 도메인이 `news.google.com`이 아닌가?
- 요약정보가 품질 기준(120자 이상, 의미 문장 2개 이상)을 충족하는가?
- 실패 항목은 재시도 이력(시도 URL, 실패 사유)이 기록되었는가?
- `원문링크`가 도메인 루트/섹션 메인 URL이 아닌가?
- `원문링크`가 소스별 허용 도메인 정책을 위반하지 않는가?
- 요약정보가 메뉴/푸터 나열이 아닌 기사 본문 중심으로 구성되었는가?

## 9) 실행용 최종 지시문
현재 시간 기준 RSS FEED 사이트를 수집/분석해 대중성이 높은 아이디어 4가지를 제안하라.
먼저 `trends_data_1차_[일시].json`(제목, 링크, 원문링크)과 `trends_data_2차_[일시].json`(제목, 링크, 요약정보)을 생성하고, 2차 데이터를 바탕으로 심층 분석하라.
Google 전체/경제/과학/AI 뉴스는 반드시 리다이렉트 해제 후 실제 기사 원문을 기준으로 요약하라.
Google 전체/경제/과학/AI 뉴스에서 `원문링크`가 `news.google.com`으로 남아 있으면 수집 완료로 판단하지 말고, 제목+언론사 기반 복원 단계를 수행해 외부 기사 URL을 확보한 뒤 요약하라.
요약정보는 품질 기준(최소 120자, 의미 문장 2개 이상)을 만족해야 하며, 실패 문구만 반환된 경우 반드시 재시도하라.
`원문링크`는 반드시 유효성 검증(도메인, 기사형 경로, 제목 키워드 일치)을 통과한 URL만 저장하라.
도메인 루트/섹션 메인/무관 페이지가 선택되면 해당 URL을 폐기하고 재탐색하라.
분석 결과를 메인 결과 파일(`YYYY년MM월DD일_HH24MMSS.md`)로 작성하고, 아이디어별 상세 실행 문서를 별도 파일(`YYYY년MM월DD일_HH24MMSS_[제안번호].md`)로 각각 작성하라.
각 아이디어에는 제안 범위의 카테고리, 실현 가능성 5점 척도, 평가 근거, 참고 RSS FEED 제목/링크/요약정보를 제공하고 주제에 맞는 전략을 수립 (예시/ 서비스 개요 및 타겟 고객, 핵심 기능, 수익 모델, 기술 스택 및 개발 단계, 리스크 및 대응 방안 등) 상세 실행 문서를 작성한다.
